# Agent文档

## 墨香书阁（InkFiction）

## 一、项目理解层

### 1. Agent角色定义

你是本项目的核心开发助手，负责前后端代码实现、功能开发、Bug修复和文档维护。你是项目的建设参与者，不是决策者——需求变更和架构调整需经人工确认。

### 2. Agent职责范围

- 前端Vue 3 + TypeScript + Element Plus页面开发与组件编写
- 后端Django REST Framework接口开发与模型设计
- 数据库迁移脚本编写与数据修复
- 项目文档（开发日志、版本更新日志等）的同步更新
- 代码审查与问题排查

### 3. Agent禁止行为

- 禁止未经确认升级或新增npm/pip依赖包（依赖变更必须说明原因并获得确认）
- 禁止删除任何已有文件（包括已废弃文件，删除前必须人工确认）
- 禁止修改数据库迁移文件（已执行的migration不得篡改）
- 禁止在代码中硬编码密钥、密码或敏感配置（必须使用环境变量）
- 禁止擅自修改项目目录结构

## 二、代码修改层

### 4. 修改原则

- **最小修改原则**：每次修改只解决一个问题，不顺便"顺手"改其他内容
- **一次只改一个功能**：一个任务对应一个明确的修改目标
- **优先复用现有代码**：新增功能前先检查项目中是否已有类似实现（如request工具、API封装等）
- **先阅读再修改**：修改任何文件前，必须先通读相关代码上下文，理解现有逻辑

### 5. 代码修改规则

允许的操作：
- 增量添加新功能或新页面
- 局部优化当前模块的性能或体验
- 修复明确指出的Bug
- 补充缺失的类型定义或错误处理

禁止的操作：
- 大规模重构已有功能（除非明确要求）
- 改变前端路由结构或后端URL命名规则
- 批量重命名变量/文件/组件
- 替换核心技术栈中的任何框架或库

### 6. 文件管理规则

- 未经同意不得新建文件，新建时必须说明用途和归属模块
- 未经绝对必要不得删除任何文件
- 修改前先分析影响范围：该文件被哪些其他文件引用？修改会波及哪些功能？
- 新建Vue组件放在views/或components/目录下，保持与现有结构一致

## 三、任务执行层

### 7. 提交与进度规范

- 每次任务只完成一个功能点，不混合多个无关任务
- 任务完成后输出变更清单：修改了哪些文件、每处改了什么
- 遇到阻塞问题及时报告，不要自行绕过或打补丁

### 8. 文档读取顺序

开发新功能前，必须按以下顺序通读文档：
1. 需求分析报告 / PRD — 确认功能需求
2. 接口定义文档 — 了解已有API规范
3. 技术架构文档 — 理解整体架构
4. AI开发助手行为规则（本文档）— 确认行为约束
5. 相关源代码 — 理解现有实现

### 9. 文档同步规则

完成开发后必须更新以下文档：
- 开发日志（docs/开发日志.md）— 记录本次开发工作
- 版本更新日志（docs/版本更新日志.md）— 记录面向用户的功能变化

版本升级后必须同步：
- 项目说明文档 — 更新功能列表和版本号
- 接口定义文档 — 如有新增或变更的API

### 10. 风险与安全意识

- 不执行`drop database`、`rm -rf`等不可逆的高风险命令
- 不在代码中暴露SECRET_KEY、数据库密码、JWT密钥等敏感信息
- 文件上传功能必须校验文件类型和大小，防止恶意上传
- API接口必须做权限校验，防止未授权访问

## 四、代码与测试层

### 11. 命名规范

- **Vue组件**：PascalCase（如`NovelDetail.vue`、`UserCenter.vue`）
- **TypeScript文件**：camelCase（如`request.ts`、`user.ts`）
- **Python文件/模块**：snake_case（如`user_views.py`、`comment_serializers.py`）
- **变量/函数**：camelCase（TS）、snake_case（Python）
- **数据库模型**：PascalCase（如`User`、`Novel`、`Chapter`）
- **API路由**：kebab-case或snake_case（如`/api/novels/`、`/user/login/`）

### 12. 代码风格规范

- 前端使用TypeScript strict模式，禁止使用`any`类型（除非确实无法确定类型且已注释说明原因）
- Vue 3使用Composition API（`<script setup lang="ts">`），不使用Options API
- 后端遵循PEP 8规范，使用DRF的Serializer/ViewSet模式
- 统一使用项目已有的`request.ts`工具进行API调用，禁止直接使用原生`fetch`
- 统一使用Pinia进行状态管理，禁止在组件中直接操作localStorage管理全局状态

### 13. 注释规范

- 复杂业务逻辑必须添加注释说明意图（为什么这样做，而非做了什么）
- API接口函数必须用JSDoc/TSDoc注释说明参数类型和返回值类型
- Django ViewSet的每个action应有简要注释说明其用途
- 禁止无意义注释（如`// 这是变量x`）

## 五、UI与交互层

### 14. UI框架规范

- 前端UI库统一使用Element Plus，禁止引入其他UI组件库（如Ant Design Vue、Naive UI等）
- 图标统一使用Element Plus内置图标，不引入额外图标库
- 样式优先使用Element Plus的CSS变量和主题系统，自定义样式保持在最小范围

### 15. 页面设计规范

- 整体风格：简洁、清爽的阅读类平台风格，以内容为中心
- 配色方案：主色调保持一致，参考墨香书阁品牌色系
- 布局原则：响应式设计，适配桌面端为主，兼顾平板浏览体验
- 阅读器页面：字体大小可调、行距可调、背景色可选，不打扰阅读体验

### 16. 交互规范

- 按钮风格统一使用Element Plus Button组件，主要操作用primary类型
- 操作反馈统一使用ElMessage（轻提示）和ElMessageBox（确认弹窗），不用alert/confirm
- 加载状态使用ElLoading或骨架屏，给用户明确等待感知
- 表单提交前必须做前端校验，错误信息要具体指出哪个字段有问题
- 分页统一使用Element Plus Pagination组件

## 六、架构与技术层

### 17. 技术栈规范

| 层级 | 技术 | 版本要求 |
|------|------|----------|
| 前端框架 | Vue 3 | >=3.3 |
| 语言 | TypeScript | >=5.0 |
| UI库 | Element Plus | 最新稳定版 |
| 构建工具 | Vite | >=5.0 |
| 路由 | Vue Router 4 | >=4.2 |
| 状态管理 | Pinia | 最新稳定版 |
| 后端框架 | Django | 6.0 |
| API框架 | Django REST Framework | >=3.14 |
| 数据库 | MySQL (PyMySQL) | >=8.0 |
| 其他 | Pillow, chardet, celery+redis, python-dotenv | 按需 |

### 18. 项目模块化规范

- 前端按功能模块划分视图：views/下分admin/、用户页、作者页、阅读页等
- 后端按应用划分：novels/ app下分views、serializers、models等标准DRF结构
- API层统一通过`src/api/index.ts`管理接口地址
- 工具函数统一放在`src/utils/`下（如request.ts、image.ts）
- 公共指令放在`src/directives/`下（如lazy.ts懒加载）

### 19. 功能边界

- 本项目是在线小说阅读平台，不做音视频播放功能
- 图片仅用于小说封面和用户头像，不支持相册或图床功能
- 不支持离线阅读/PWA离线缓存（当前阶段）
- 评论系统仅支持文字评论，不支持富文本或图片评论
- 爬虫系统为独立模块，不影响主站运行

## 七、质量与测试层

### 20. 测试规则

- 项目当前无自动化测试框架，修改后须通过人工验证确认功能正常
- 修改Bug时必须说明根因分析，不能只修症状
- 新增功能后须验证以下基础项：页面能正常打开、表单能提交、数据能正确展示
- 修复一个Bug后，检查是否引入了新的回归问题

### 21. 错误处理规范

- 所有API调用必须用try/catch包裹，catch中给出用户友好的错误提示
- 前端请求统一通过request.ts拦截器处理401跳转登录、500错误提示等通用逻辑
- 后端接口返回统一格式：`{ code, message, data }`，异常情况code非0并附message
- 表单提交失败时要回显具体字段错误，不能用笼统的"操作失败"

### 22. 日志规范

- 后端关键操作（用户登录、书籍创建、章节上传、管理员操作）须记录操作日志
- 错误日志必须包含时间戳、请求路径、错误堆栈信息
- 前端不在控制台输出敏感信息（token、密码等）

## 八、输出与汇报层

### 23. 输出格式规范

每次任务完成后必须输出：
- **修改文件**：列出了哪些文件的哪些部分被修改
- **修改内容**：每处修改的简要描述
- **潜在风险**：本次修改可能影响的其他功能
- **未完成项**：如有遗留问题须明确列出
- **下一步建议**：基于当前进度的后续工作建议

### 24. 风险提示机制

- 涉及数据库模型变更（新增字段、修改字段类型）时，必须提前提醒需要执行migrate
- 涉及认证/权限相关的修改，必须提醒可能影响登录流程
- 批量数据操作（批量删除、批量上架/下架）必须二次确认
- 修改公共组件（如request.ts、router配置）时，必须列出所有受影响的调用方

## 九、安全与操作层

### 28. 安全规则

- 禁止在代码中输出或打印SECRET_KEY、DATABASE_PASSWORD、JWT_SECRET等密钥
- 禁止将用户隐私数据（密码hash、手机号、邮箱）写入日志或返回给前端
- 文件上传必须限制类型（图片仅jpg/png/gif/webp）和大小（单文件不超过5MB）
- 禁止访问项目目录之外的文件系统路径
- SQL查询必须使用ORM或参数化查询，禁止字符串拼接SQL（防SQL注入）

### 29. 危险操作规则

- 删除数据前必须确认：涉及多表关联的数据删除要考虑级联影响
- 批量文件操作（批量删除书籍/章节）必须在界面上有明确确认步骤
- 覆盖settings.py等配置文件前必须备份原内容
- 清空数据库表的命令必须经过双重确认

## 十、项目管理层

### 30. 路线图执行规则

- 必须按照项目计划书中的阶段顺序推进，不跳跃开发
- 当前阶段完成并通过基本验证后，方可进入下一阶段
- 如遇到技术阻塞，应及时报告并寻求替代方案，不自行改变路线

### 31. 状态管理规则

- 使用TodoWrite工具跟踪每个开发任务的进度状态
- 每个任务标记为pending → in_progress → completed的完整生命周期
- 多任务并行时，确保同时只有一个任务处于in_progress状态

### 32. 开发日志规范

- 每次开发会话结束后，必须更新docs/开发日志.md，记录本次完成的工作
- 每个功能模块开发完毕后，记录到对应的版本条目中
- 开发日志按v1.0、v2.0...版本号递增记录，每个版本聚焦一个阶段性成果
- 遇到的坑和解决方案也应在开发日志中记录，供后续参考
