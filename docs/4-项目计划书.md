## 一、总体计划

本项目旨在构建一个企业级高品质小说阅读平台——墨香书阁（Mo Xiang Book Pavilion）V3.0.0，整合用户阅读体验、社交互动功能和管理后台三大核心模块，为文学爱好者提供专业级的在线阅读服务。项目采用前后端分离架构，前端使用Vue 3 + TypeScript + Vite + Element Plus构建11个用户端页面和5个管理端页面，后端基于Python 3.8 + Django 4.2 + DRF开发9个核心业务模块，数据库采用MySQL存储81本小说、2381章节数据及12个分类体系。

项目预计开发周期为2周，投入1-2人全栈开发团队。开发遵循敏捷迭代模式，分五个里程碑递进实施：从基础框架搭建起步，逐步实现核心阅读功能、社交互动系统（关注作者+评分+签到会员）、个人中心和管理后台，最终完成全量交付和系统优化。以PRD.md为需求基准，Agent.md为开发规范，确保过程有序可控、交付质量可靠。

## 二、项目范围说明书

### 2.1 范围基准

项目交付范围包括以下核心部分：

- **前端用户系统**：11个页面（首页、书库、详情页、排行榜、搜索、作者主页、我的关注、签到页、会员页、个人中心、404页面）
- **后台管理系统**：5个页面（图书管理、分类管理、订单管理、签到管理、粉丝列表）
- **后端API服务**：9个核心模块（小说管理API、搜索API、用户认证API、收藏进度API、关注作者API(6个接口)、评分API(3个接口)、签到会员API、管理后台API）
- **数据库设计**：15张数据表（原有12张 + 新增UserFollow/NovelRating/Checkin+Membership共3张）
- **数据规模**：81本小说 / 2381章节 / 12个分类
- **认证机制**：Session-based Cookie withCredentials
- **项目文档**：14份完整文档

### 2.2 项目范围管理计划

范围变更由项目负责人统一评估和批准。新增功能需求需提交变更申请，评估对现有模块的影响范围、工作量和进度影响，经确认后方可实施。每完成一个里程碑后进行范围复核，确认已交付功能与需求规格说明书一致，无遗漏或超出范围的情况。对于非核心需求的变更，优先安排在后续版本迭代中实现，避免影响当前版本的核心目标。

### 7.2 风险防范措施

| 编号 | 防范措施 |
|------|---------|

### 9.2 版本管理规则

**语义化版本号规范（Semantic Versioning）**：

- **主版本号（MAJOR）**：不兼容的API变更、重大架构重构、数据库Schema破坏性变更
  - 示例：v2.0.0 → v3.0.0（新增3张社交表+9个社交API+企业级UI重构）
- **次版本号（MINOR）**：向下兼容的功能新增、新页面/新模块上线
  - 示例：v3.0.0 → v3.1.0（新增消息通知模块）
- **修订号（PATCH）**：向下兼容的问题修复、Bug修复、小的优化改进
  - 示例：v3.0.0 → v3.0.1（修复评分平均分计算精度Bug）

**分支管理策略**：

```
main（生产分支，受保护）
  ├── develop（开发分支，日常开发在此进行）
  │     ├── feature/social（功能分支，如feature/follow-rating-checkin）
  │     ├── feature/ui-theme（功能分支，如feature/deep-blue-theme）
  │     └── fix/xxx（修复分支，如fix/session-auth-issue）
  └── release/x.x.x（预发布分支，版本发布前的最终测试）
```

**标签命名规范**：

- 正式版本：`v{MAJOR}.{MINOR}.{PATCH}` （如 v3.0.0）
- 预发布版本：`v{MAJOR}.{MINOR}.{PATCH}-rc.{N}` （如 v3.0.0-rc.1）
- 开发版本：`v{MAJOR}.{MINOR}.{PATCH}-dev.{YYYYMMDD}` （如 v3.0.0-dev.20260607）

每次版本发布时：
1. 从develop分支创建release分支
2. 在release分支上进行最终测试和文档更新
3. 合并release分支到main分支
4. 在main分支上创建Git Tag
5. 同步更新CHANGELOG.md和14份项目文档中的版本号为v3.0.0
6. 将main分支的最新代码部署到生产环境

### 9.3 变更控制

**代码变更流程**：

1. **变更申请**：开发者在本地feature分支进行开发，完成后提交Pull Request/Merge Request
2. **代码审查**：项目负责人或另一位开发者进行Code Review，检查代码质量、逻辑正确性和安全性
3. **自动化检查**：CI流水线（如有）自动执行linting、类型检查和单元测试
4. **合并批准**：审查通过后合并到develop分支
5. **测试验证**：在develop分支或release分支上进行集成测试（重点关注社交功能联调）
6. **发布合并**：测试通过后合并到main分支并打Tag发布

**文档变更流程**：

1. 文档变更随对应功能的代码一起提交，保持文档与代码同步
2. 每次文档修改在文件头部或尾部注明修改日期、修改人和修改内容摘要
3. 重要文档变更（如需求规格、接口定义、数据库Schema）需经过评审
4. 版本发布前全面检查14份文档，确保版本号和功能描述与实际代码一致

**数据库变更流程**：

1. 使用Django makemigrations生成迁移文件（重点关注UserFollow/NovelRating/Checkin表）
2. 审查迁移文件的SQL语句，确保无数据丢失风险（特别是外键和级联删除）
3. 在测试环境先执行migrate验证
4. 生产环境迁移前备份数据库
5. 执行migrate并验证数据完整性（检查15张表数据均正常）
6. 记录迁移操作到操作日志

**紧急变更（Hotfix）流程**：

1. 从main分支创建hotfix分支
2. 快速修复问题并进行最小化测试
3. 合并hotfix到main和develop分支
4. 立即打补丁版本Tag（如v3.0.1）
5. 补充更新相关文档和CHANGELOG

### 9.4 备份策略

**代码备份策略**：

- **主备份**：GitHub远程仓库作为代码的主备份，每次push自动同步
- **本地备份**：每位开发者在本地保留完整的Git克隆副本
- **分支备份**：重要里程碑版本（M1-M5）在远程仓库创建永久存档分支
- **备份频率**：代码变更实时通过Git push备份，至少每日push一次

**数据库备份策略**：

- **自动备份**：使用mysqldump每日凌晨2点自动执行全量备份（含15张表全部数据），保留最近7天的备份文件
- **备份存储**：备份文件压缩后存储在独立磁盘分区或云存储，与原始数据隔离
- **备份验证**：每周随机抽取一个备份文件进行恢复测试，确保备份可用（重点验证3张新表数据完整性）
- **迁移前备份**：每次执行数据库migration前强制执行一次手动备份
- **备份脚本**：编写Shell/Python脚本自动化备份流程，记录备份日志

**媒体资源备份策略**：

- **封面图片**：定期同步media/covers/目录到云存储或异地备份
- **静态资源**：前端dist/打包产物在CI/CD中自动归档
- **用户上传内容**：如有用户头像等上传功能，定期备份uploads/目录

**文档备份策略**：

- **版本控制**：14份项目文档纳入Git版本管理，每次修改都有历史记录
- **导出备份**：每个里程碑节点将docs/目录导出为ZIP归档包单独保存
- **多云备份**：重要文档同时在GitHub和本地磁盘保留副本

**灾难恢复预案**：

1. **代码丢失**：从GitHub远程仓库clone恢复，最多丢失当日未push的代码
2. **数据库损坏**：从最近的mysqldump备份恢复（含15张表），最多丢失1天数据
3. **服务器故障**：在新服务器上重新部署，从Git拉取代码、从备份恢复数据库
4. **人为误操作**：利用Git的reflog和reset功能恢复误删的代码或分支
