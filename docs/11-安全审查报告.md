## 二、已妥善处理（9 项）

| 检查项 | 涉及文件 | 处理方式 |
|--------|---------|---------|
| Session + Cookie 认证 | 前端 utils/request.ts / 后端 auth.py | 使用 Session 认证（Cookie withCredentials），前端 request.ts 全局配置 `withCredentials: true`；后端自定义 `CsrfExemptSessionAuthentication` 认证类，兼顾 CSRF 豁免与会话安全性 |
| 自定义认证后端隔离 | 后端 models.py (AdminUser) / auth.py | AdminUser 为独立模型（非 Django User），通过自定义 Authentication Backend 实现 `authenticate` 方法，管理员与普通用户账号体系完全隔离，避免权限混淆 |
| DRF 权限控制 | 后端 follow_views.py / rating_views.py / checkin_views.py | 新增模块均正确设置 `permission_classes = [IsAuthenticated]`，未登录用户无法调用关注、评分、签到、会员等敏感接口 |
| ORM 参数化查询防 SQL 注入 | 全部后端 views 文件 | 所有数据库操作均通过 Django ORM 执行（`filter`、`get_or_create`、`update_or_create`、`annotate`、`aggregate` 等），ORM 自动使用参数化查询，从根本上杜绝 SQL 注入风险 |
| XSS 防护（Vue 模板自动转义） | 前端全部 Vue 组件 | Vue 3 模板默认对插值表达式 `{{ }}` 进行 HTML 实体转义；评论内容使用 `{{ c.content }}` 渲染（非 v-html）；NovelDetail.vue 评论区和 Home.vue 公告标题均使用文本插值，无 `v-html` 使用风险 |
| 密码安全存储 | 后端 models.py (AdminUser) / Django User 模型 | AdminUser 的 `set_password` 调用 Django 内置 `make_password`（PBKDF2 算法）；Django User 模型密码同样由 Django Auth 系统自动哈希存储，不可逆 |
| .env 文件保护 | 项目根目录 .gitignore | `.env` 文件已加入 `.gitignore`，数据库密码、密钥等敏感配置不会随代码提交至仓库 |
| 无硬编码密钥 | 全项目代码扫描 | 经全项目搜索确认：代码中不存在硬编码的数据库密码、API Secret Key、JWT 密钥等敏感信息；所有密钥均通过环境变量或 Django settings 引用 |
| 敏感接口参数校验 | rating_views.py / follow_views.py / checkin_views.py | rating submit 接口四层校验（存在性→格式合法性→范围1–5→小说存在性）；follow 接口校验 author_name 非空；checkin 接口先查重后写入，防止重复签到 |

## 三、需关注（5 项）

| 检查项 | 风险等级 | 说明 |
|--------|---------|------|
| API 频率限制缺失 | 中 | 登录、注册、签到、评分、关注等接口未设置请求频率限制（throttling）。签到接口 (`do_checkin`) 尤其容易被脚本批量调用刷取虚拟币；评分接口 (`submit`) 可能被恶意刷分。建议引入 DRF 内置 `throttle_classes` 或第三方库（如 `django-ratelimit`）按接口分级限流 |
| followers 接口数据暴露范围过大 | 中 | `follow_views.py` 的 `followers` 接口在未传 `author_name` 参数时返回**全部用户的关注关系数据**（含 username + author_name + created_at）。当前仅限制登录用户可访问，但普通用户仍可遍历全站社交关系图谱，存在隐私泄露风险 |
| CORS 配置需生产环境收紧 | 低 | 开发阶段 CORS 配置可能允许较广泛的来源访问（`CORS_ALLOWED_ORIGINS`）。部署到生产环境前必须收窄为仅允许正式域名，防止未授权站点发起跨域请求携带用户凭证 |
| DEBUG=True 生产环境风险 | 高 | 若 Django 的 `DEBUG` 模式在生产环境开启，将泄露详细错误堆栈、配置信息、已安装中间件列表等敏感数据。部署前必须关闭 DEBUG 并配置 `ALLOWED_HOSTS` |
| 签到接口缺少并发控制 | 中 | `do_checkin` 接口采用"先查后写"模式（先 SELECT 查询今日是否存在，再 INSERT 创建记录），在高并发场景下存在竞态条件（TOCTOU）：两个几乎同时到达的请求可能同时通过 exists() 检查，导致重复签到 |

> 📝 **风险等级定义**：高=必须修复；中=建议修复；低=可接受但需监控。

## 四、建议修复（5 项）

| 问题 | 涉及文件 | 说明 |
|------|---------|------|
| 添加 API 请求频率限制 | 后端 settings.py / follow_views.py / rating_views.py / checkin_views.py | 使用 DRF 的 `throttle_classes` 或自定义中间件，对不同接口实施差异化限流策略：登录接口每分钟 5 次、注册每小时 10 次、签到接口每天 1 次、评分接口每分钟 10 次、关注接口每分钟 20 次。签到和评分作为用户行为接口，建议使用 `ScopedRateThrottle` 按 user 维度限流 |
| followers 接口增加权限粒度控制 | follow_views.py | 对不传 `author_name` 参数的全量查询增加 `IsAdminUser` 权限检查；普通用户只能查询指定作者的粉丝列表。修改 `followers` action 方法，在 `author_name` 为空时判断 `request.user.is_staff` |
| 签到接口改用数据库级唯一约束防并发 | checkin_views.py / models.py | 当前 CheckIn 模型的 `unique_together = ['user', 'check_date']` 已提供数据库级唯一约束，但 `do_checkin` 直接调用 `objects.create()` 在冲突时会抛出 IntegrityError 而非返回友好提示。**建议**改用 `get_or_create` 替代 `exists()` + `create()` 两步操作，利用数据库唯一约束保证原子性，捕获 `IntegrityError` 返回"今日已签到"提示 |
| 关闭 DEBUG 模型并配置生产安全项 | Django settings.py | 生产环境设置 `DEBUG = False`，配置 `ALLOWED_HOSTS = ['你的域名']`，配置自定义 404/500 错误页面，设置 `SECURE_SSL_REDIRECT = True`、`SESSION_COOKIE_SECURE = True`、`CSRF_COOKIE_SECURE = True`。同时配置 `LOGGING` 将错误输出到文件而非标准输出 |
| create_order 支付流程加固 | checkin_views.py | 当前 `create_order` 创建订单后直接设为 `paid` 状态（模拟支付）。正式商用时需：(1) 移除直接赋值 paid 的逻辑；(2) 对接支付渠道回调通知接口；(3) 回调接口必须验证签名/金额/订单号一致性，防止伪造支付回调；(4) 订单金额必须从服务端 PLAN_CONFIG 读取，不允许客户端传入 amount 参数 |

## 五、总体评价

墨香书阁 V3.0.0 安全意识较好，核心安全路径均有基础防护措施。认证体系（Session + Cookie withCredentials + 自定义认证后端）、授权机制（DRF IsAuthenticated/IsAdminUser 分级权限）、注入防护（ORM 参数化查询全覆盖）、XSS 防护（Vue 模板自动转义 + 无 v-html 滥用）、密码安全（Django make_password 哈希存储）、配置安全（.env gitignore + 无硬编码密钥）等关键环节均已正确实现，构成了完整的安全基线。

**v3.0.0 新增功能模块安全评价**：
- **关注作者（follow_views.py）**：6 个 API 均设置了 IsAuthenticated 权限，follow/unfold 使用 get_or_create/delete 保证幂等性。主要风险点在于 followers 接口的数据暴露范围需收窄
- **小说评分（rating_views.py）**：submit 接口的四层参数校验是亮点（存在性→格式→范围→业务存在性），update_or_create 利用 unique_together 保证幂等。stat 和 my_rating 接口的公开/鉴权划分合理
- **签到系统（checkin_views.py）**：实现了登录检查、重复签到检查、AdminUser 隔离三重防护。_calc_consecutive 有 365 天上限防止无限循环。主要改进方向是并发竞态条件修复（改用 get_or_create）
- **会员充值（checkin_views.py MembershipViewSet）**：当前为模拟支付模式，不存在真实资金风险。PLAN_CONFIG 服务端硬编码价格防止客户端篡改金额。正式商用前需完整对接支付渠道

主要改进方向集中在生产环境加固层面：API 频率限制的系统性引入、签到接口并发控制修复、followers 数据暴露范围收窄、DEBUG 模式的关闭以及 HTTPS/Cookie 安全配置。对于会员充值功能，重点需要在商用前完成支付回调对接和签名验证。

**综合风险等级：🟡 中低。（修复上述高/中优先级建议项后可达生产标准。）**
