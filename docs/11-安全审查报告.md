## 二、已妥善处理（7项）

| 检查项 | 涉及文件 | 处理方式 |
|--------|---------|---------|
| 密码加密存储 | 后端 user_views.py / models.py | 用户密码使用 bcrypt 算法进行单向哈希存储，不可逆，即使数据库泄露也无法还原明文密码 |
| JWT Token 认证 | 后端 auth.py | 基于 PyJWT 实现 Token 签发与验证，有效期 24 小时，Token 中不含敏感信息（如密码），仅携带用户 ID 和角色 |
| CORS 跨域配置 | Django settings.py / django-cors-headers | 通过 django-cors-headers 中间件统一配置允许的跨域来源，避免浏览器同源策略被绕过 |
| 角色权限分离 | 后端 auth.py + 前端 router/index.ts | 四级角色体系（普通用户/VIP会员/作者/管理员），后端 permission 类控制 API 访问，前端路由守卫控制页面访问，双重防护 |
| ORM 参数化查询防 SQL 注入 | 全部后端 views 文件 | 所有数据库操作均通过 Django ORM 执行，ORM 自动使用参数化查询，从根本上杜绝 SQL 注入风险 |
| CSRF 防护 | Django settings.py | 启用 Django 内置 CSRF 中间件，所有状态修改请求均需携带 CSRF Token，防止跨站请求伪造攻击 |
| 文件上传限制 | 后端 author_views.py (upload_cover / TXT上传) | 上传文件类型白名单校验（仅允许图片格式/TXT）+ 文件大小上限限制，防止恶意文件上传 |

## 三、需关注（4项）

| 检查项 | 风险等级 | 说明 |
|--------|---------|------|
| XSS 防护需加强 | 中 | 评论内容等用户输入在渲染到页面时，若未做 HTML 转义处理，可能存在存储型 XSS 风险。建议对用户生成内容（UGC）统一实施输出转义或使用 CSP 策略加固 |
| CORS 配置较宽松 | 低 | 当前 CORS 配置可能允许较广泛的来源访问。开发阶段可接受，但生产环境应收紧为仅允许正式域名，防止未授权站点发起跨域请求 |
| API 频率限制缺失 | 中 | 登录、注册、搜索等公开接口未设置请求频率限制（throttling），存在被暴力破解或爬虫滥用的风险。建议引入 DRF 内置限流类或第三方库（如 django-ratelimit） |
| DEBUG=True 生产环境风险 | 高 | Django 的 DEBUG 模式在生产环境开启会泄露详细错误堆栈、配置信息等敏感数据。部署前必须关闭 DEBUG 并配置 ALLOWED_HOSTS |

> 📝 **风险等级定义**：高=必须修复；中=建议修复；低=可接受但需监控。

## 四、建议修复（4项）

| 问题 | 涉及位置 | 说明 |
|------|---------|------|
| 添加 XSS 过滤中间件 | 后端全局 / 前端渲染层 | 对所有用户提交的内容（评论、昵称、简介等）在输出前进行 HTML 实体转义（`<` → `&lt;`, `>` → `&gt;` 等），或在前端使用 `v-text` 替代 `v-html` 渲染 UGC 内容。同时建议添加 Content-Security-Policy 响应头 |
| 添加 API 请求频率限制 | 后端 settings.py / views.py | 使用 DRF 的 `throttle_classes` 或自定义中间件，对登录接口（如每分钟 5 次）、注册接口（每小时 10 次）、搜索接口（每分钟 30 次）分别设置限流策略，防止暴力破解和资源滥用 |
| 关闭 DEBUG 模型 | Django settings.py | 生产环境设置 `DEBUG = False`，配置 `ALLOWED_HOSTS = ['你的域名']`，并配置自定义 404/500 错误页面，避免泄露服务器信息 |
| 添加 HTTPS 强制跳转 | Django settings.py / Web 服务器（Nginx/Apache） | 生产环境全站启用 HTTPS，配置 `SECURE_SSL_REDIRECT = True` 和 `SESSION_COOKIE_SECURE = True`，同时在 Web 服务器层面配置 HTTP → 301 重定向至 HTTPS，确保传输链路加密 |

## 五、总体评价

墨香书阁项目安全意识较好，核心安全路径均有基础防护措施。认证（JWT + bcrypt）、授权（四级角色分离）、注入防护（ORM 参数化查询）、CSRF 防护、文件上传限制等关键环节均已正确实现，构成了完整的安全基线。

**v2.0.0新增功能安全评价**：
- **签到系统**：已实现登录认证和重复签到检查等基础防护，但缺少防刷机制和并发控制，存在被脚本批量刷积分的风险
- **VIP充值系统**：当前为模拟支付模式，不存在真实资金风险，金额由服务端校验防止篡改；正式商用前需对接真实支付渠道并实施完整的安全方案
- **首页聚合API**：作为公开接口返回数据经过过滤不包含敏感字段，安全性可接受
- **爬虫脚本**：实现了User-Agent伪装和错误处理，但缺少速率限制，可能存在合规性风险

主要改进方向集中在生产环境加固层面：XSS 输出转义的系统性覆盖、API 频率限制的补充、DEBUG 模式的关闭以及 HTTPS 的强制启用。对于新增功能，重点需要加强签到防刷、支付安全和爬虫合规性三方面的措施。

**综合风险等级：🟡 中低。（修复上述建议项后可达生产标准。）**
