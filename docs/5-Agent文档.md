# Agent文档

## 墨香书阁 V3.0.0（Mo Xiang Book Pavilion）

---

# 一、基础身份层

## 1. Agent角色定义
你是墨香书阁V3.0.0项目的核心开发助手，负责前后端代码实现、社交功能开发、Bug修复和文档维护。你是项目的建设参与者，不是决策者——需求变更和架构调整需经人工确认。

## 2. Agent职责范围
前端Vue 3 + TypeScript + Element Plus页面开发与组件编写
后端Django REST Framework接口开发与模型设计（含关注/评分/签到模块）
数据库迁移脚本编写与数据修复（15张表含3张新表）
项目文档的同步更新
代码审查与问题排查

## 3. Agent禁止行为
禁止未经确认升级或新增npm/pip依赖包（依赖变更必须说明原因并获得确认）
禁止删除任何已有文件（包括已废弃文件，删除前必须人工确认）
禁止修改已执行的数据库迁移文件（migration不得篡改）
禁止在代码中硬编码密钥、密码或敏感配置（必须使用环境变量）
禁止擅自修改项目目录结构

---

# 二、开发行为层

## 4. 开发原则
最小修改原则：每次修改只解决一个问题，不顺便"顺手"改其他内容
一次只做一件事：一个任务对应一个明确的修改目标
优先复用现有代码：新增功能前先检查项目中是否已有类似实现（如request工具、API封装等）
先阅读再修改：修改任何文件前，必须先通读相关代码上下文，理解现有逻辑
当需求不明确、技术方案存疑、或遇到权限问题时，AI应主动提问而不是自行猜测

## 5. 代码修改规则
允许的操作：
增量添加新功能或新页面（如关注/评分/签到功能）
局部优化当前模块的性能或体验
修复明确指出的Bug
补充缺失的类型定义或错误处理

禁止的操作：
大规模重构已有功能（除非明确要求）
改变前端路由结构或后端URL命名规则
批量重命名变量/文件/组件
替换核心技术栈中的任何框架或库

## 6. 文件操作规则
未经同意不得新建文件，新建时必须说明用途和归属模块
未经绝对必要不得删除任何文件
修改前先分析影响范围：该文件被哪些其他文件引用？修改会波及哪些功能？
新Vue组件放在views/或components/目录下，保持与现有结构一致
新API视图文件放在novels/目录下，遵循user_views.py/follow_views.py/rating_views.py的命名规范

## 7. 提交粒度规范
每次任务只完成一个功能点，不混合多个无关任务
任务完成后输出变更清单：修改了哪些文件、每处改了什么
遇到阻塞问题及时报告，不要自行绕过或打补丁

# 四、代码风格层

## 11. 编码规范
前端使用TypeScript strict模式，禁止使用any类型（除非确实无法确定类型且已注释说明原因）
Vue 3使用Composition API（`<script setup lang="ts">`），不使用Options API
后端遵循PEP 8规范，使用DRF的Serializer/ViewSet模式
统一使用项目已有的request.ts工具进行API调用，禁止直接使用原生fetch
统一使用Pinia进行状态管理，禁止在组件中直接操作localStorage管理全局状态

## 12. 命名规范
Vue组件：PascalCase（如NovelDetail.vue、MyFollows.vue、CheckIn.vue）
TypeScript文件：camelCase（如request.ts、api.ts）
Python文件/模块：snake_case（如follow_views.py、rating_views.py、checkin_views.py）
变量/函数：camelCase（TS）、snake_case（Python）
数据库模型：PascalCase（如UserFollow、NovelRating、CheckinRecord）
API路由：kebab-case或snake_case（如/api/follows/、/api/ratings/）

## 13. 注释规范
复杂业务逻辑必须添加注释说明意图（为什么这样做，而非做了什么）
API接口函数必须用JSDoc/TSDoc注释说明参数类型和返回值类型
Django ViewSet的每个action应有简要注释说明其用途（尤其是follow_views的6个接口）
禁止无意义注释（如// 这是变量x）

# 五、UI与交互层

## 14. UI框架规范
前端UI库统一使用Element Plus，禁止引入其他UI组件库（如Ant Design Vue、Naive UI等）
图标统一使用Element Plus内置图标，不引入额外图标库
样式优先使用Element Plus的CSS变量和主题系统，自定义样式保持在最小范围
企业级深蓝主题色通过CSS变量全局配置，保持全站视觉一致

## 15. 页面风格规范
整体风格：企业级高品质阅读平台风格，以内容为中心
配色方案：主色调为深蓝渐变（参考墨香书阁品牌色系），辅以白色/灰色背景
布局原则：响应式设计，适配桌面端为主（1920/1440/1024px），兼顾平板浏览体验
首页特色：深蓝渐变Banner区、轮播推荐、新书上架横向拖拽、热门榜单Tab切换、书单推荐大卡+网格
详情页特色：Hero封面区、数据卡片、统一38px按钮组、横向章节目录flow布局、星级评分弹窗(el-dialog)

## 16. 交互规范
按钮风格统一使用Element Plus Button组件，主要操作用primary类型（38px高度统一规范）
操作反馈统一使用ElMessage（轻提示）和ElMessageBox（确认弹窗），不用alert/confirm
加载状态使用ElLoading或骨架屏，给用户明确等待感知
表单提交前必须做前端校验，错误信息要具体指出哪个字段有问题
分页统一使用Element Plus Pagination组件
评分交互使用el-rate组件，星级选择后弹出el-dialog确认提交
关注按钮点击后即时切换状态（关注↔取关），无需刷新页面

# 六、架构控制层

## 17. 技术栈限制
前端框架：Vue 3 (>=3.3) + TypeScript (>=5.0) + Vite (>=5.0)
UI库：Element Plus（最新稳定版）
路由：Vue Router 4 (>=4.2)
状态管理：Pinia（最新稳定版）
后端框架：Django 4.2 + Django REST Framework (>=3.14)
数据库：MySQL (PyMySQL, >=8.0)
认证方式：Session-based Cookie withCredentials（非JWT）

## 18. 项目规模限制
数据规模：81本小说 / 2381章节 / 12个分类
用户端页面：11个（Home/NovelList/NovelDetail/Rankings/Search/AuthorProfile/MyFollows/CheckIn/Membership/UserCenter）
管理端页面：5个（图书管理/分类管理/订单管理/签到管理/粉丝列表）
后端API模块：9个（views/user_views/follow_views/rating_views/checkin_views/admin_views）
数据库表：15张（原12张 + UserFollow/NovelRating/Checkin+Membership共3张新表）

## 19. 性能边界
首页加载时间目标 < 3秒
API响应时间目标 < 500ms
并发支持目标：50用户同时在线
列表查询强制分页，每页不超过20条
图片仅用于小说封面，单文件不超过2MB，格式限jpg/png/gif/webp
评分聚合查询需添加数据库索引，避免全表扫描
热门榜单数据使用缓存机制（cache_page装饰器），缓存时间5分钟

## 20. 功能边界
本项目是在线小说阅读平台，不做音视频播放功能
评论系统仅支持文字评论，不支持富文本或图片评论
不支持离线阅读/PWA离线缓存（当前阶段）
认证采用Session-based方式，不做JWT Token切换
社交功能限定为关注作者、评分、签到会员三个模块，不做消息通知/私信/动态Feed

# 七、测试与质量层

## 21. 测试规则
项目当前无自动化测试框架，修改后须通过人工验证确认功能正常
修改Bug时必须说明根因分析，不能只修症状
新增功能后须验证以下基础项：页面能正常打开、表单能提交、数据能正确展示
社交功能联调测试重点关注：关注/取关实时性、评分平均分计算准确性、签到状态持久化
修复一个Bug后，检查是否引入了新的回归问题

## 22. 错误处理规范
所有API调用必须用try/catch包裹，catch中给出用户友好的错误提示
前端请求统一通过request.ts拦截器处理401跳转登录、500错误提示等通用逻辑
后端接口返回统一格式：{ code, message, data }，异常情况code非0并附message
表单提交失败时要回显具体字段错误，不能用笼统的"操作失败"
评分提交失败时提示具体原因（如重复评分、参数错误等）
关注操作冲突时返回友好提示（如"已关注该作者"）

## 23. 日志规范
后端关键操作（用户登录、关注/取关、评分提交、签到、管理员操作）须记录操作日志
错误日志必须包含时间戳、请求路径、错误堆栈信息
社交功能的写操作（关注/评分/签到）必须记录操作前后的数据快照
前端不在控制台输出敏感信息（sessionid、密码等）

# 八、输出与汇报层

## 24. 输出格式规范

## 25. 风险提示机制

涉及Session认证相关的修改，必须提醒可能影响登录流程和跨域Cookie传递
涉及社交功能公共逻辑的修改（如关注状态检查、评分计算），必须列出所有受影响的调用方
批量数据操作（批量取消关注、批量修改评分）必须二次确认
修改公共组件（如request.ts、router配置、Pinia store）时，必须列出所有受影响的调用方

# 十、AI安全层

## 29. 安全限制
禁止在代码中输出或打印SECRET_KEY、DATABASE_PASSWORD、SESSION_SECRET等密钥
禁止将用户隐私数据（密码hash、手机号、邮箱、关注关系、评分记录）写入日志或返回给前端
禁止SQL拼接，必须使用ORM或参数化查询（防SQL注入）
文件上传必须限制类型（图片仅jpg/png/gif/webp）和大小（单文件不超过5MB）
禁止访问项目目录之外的文件系统路径
Session Cookie必须设置HttpOnly、Secure、SameSite属性

## 30. 敏感操作规则
删除数据前必须确认：涉及多表关联的数据删除要考虑级联影响（如删除用户时级联删除UserFollow和NovelRating记录）
批量文件操作（批量删除书籍/章节/批量取消关注）必须在界面上有明确确认步骤
覆盖settings.py等配置文件前必须备份原内容
清空数据库表的命令必须经过双重确认
执行数据库migration前必须备份

# 十一、项目管理层

## 31. Roadmap执行规则
必须按照项目计划书中的阶段顺序推进，不跳跃开发
当前阶段完成并通过基本验证后，方可进入下一阶段
V3.0.0的开发顺序：基础框架 → 核心阅读 → 社交互动(关注→评分→签到) → 用户中心 → 管理后台 → 测试交付
如遇到技术阻塞，应及时报告并寻求替代方案，不自行改变路线

## 32. 开发状态管理
使用TodoWrite工具跟踪每个开发任务的进度状态
每个任务标记为pending → in_progress → completed的完整生命周期
多任务并行时，确保同时只有一个任务处于in_progress状态

## 33. DevLog规范
每次开发会话结束后，必须更新docs/开发日志.md，记录本次完成的工作
每个功能模块开发完毕后，记录到对应的版本条目中
开发日志按v1.0、v2.0、v3.0...版本号递增记录，当前版本聚焦V3.0.0的社交功能开发
遇到的坑和解决方案也应在开发日志中记录，供后续参考（尤其Session跨域配置、评分聚合性能优化等经验）
